# review 1
검토 결과, 이번 실행 결과는 **단순히 색상이 덜 바뀐 수준이 아니라, 수정 패치가 일부 탭의 UI 생성을 깨뜨린 상태**입니다. 특히 `inputBg` 오류 때문에 Options 탭 생성이 중단되고, 그 이후 일부 Edit Dialog 탭들이 **빈 화면으로 남는 연쇄 문제가 발생**한 것으로 보입니다.

## 1. 즉시 수정해야 할 런타임 오류

에러의 직접 원인은 명확합니다.

```text
"inputBg"은(는) 인식할 수 없는 필드 이름입니다.
오류 발생: FlightDataDashboard/buildEditTabOptions (6052번 라인)
'BackgroundColor', tOpt.inputBg
```

즉, `buildEditTabOptions()`에서 `tOpt.inputBg`를 사용했지만, `getLightTheme()`가 반환하는 theme struct 안에는 `inputBg` 필드가 없습니다. 기존 코드의 `getLightTheme()`에는 `windowBg`, `surfaceBg`, `surfaceAltBg`, `headerBg`, `borderColor`, `textPrimary`, `textSecondary`, `textMuted`, `accentBlue` 등은 정의되어 있지만, `inputBg`는 확인되지 않습니다. 

이 오류 때문에 `openEditDialog()` 실행 중 `buildEditTabOptions(tabOpts)`에서 멈추고, 이후 refresh/theme 적용/탭 내부 생성이 끝까지 수행되지 않습니다. 따라서 일부 캡처에서 Options, Plot Manager, Export 탭이 **흰 빈 화면** 또는 **내용 없는 패널**로 보이는 것은 단순 스타일 문제가 아니라 **UI build 실패의 결과**로 판단됩니다. `buildEditTabOptions()` 자체는 Options 탭의 핵심 UI를 만드는 함수이므로 여기서 예외가 나면 Edit Dialog 전체의 후속 초기화가 불완전해질 수 있습니다. 

**우선 조치:** `getLightTheme()`에 최소한 다음 필드를 추가해야 합니다.

```matlab
t.inputBg = [1.00 1.00 1.00];
t.inputFg = t.textPrimary;
t.inputBorder = t.borderColor;
```

또는 더 안전하게 `buildEditTabOptions()`에서 `tOpt.inputBg`를 `tOpt.surfaceBg`로 바꿔야 합니다. 현재처럼 theme 필드를 함수별로 임의 추가하면 같은 오류가 반복됩니다.

## 2. PNG별 문제 검토

### 401 / Project 탭

Project 탭은 여전히 전체 본문이 거의 검은색입니다. 사용자가 처음부터 요청한 “샘플.png에는 검은색 배경이 없음” 조건을 전혀 만족하지 못합니다. 글자는 일부 흰색 또는 어두운 파란색이고, 링크성 텍스트는 검은 배경 위에서 거의 묻힙니다. `저장된 preset` dropdown도 검은 배경과 회색 테두리라 MATLAB 기본 dark control처럼 보입니다.

문제는 단순 색상값 하나가 아니라, **Edit Dialog의 tab content parent 자체가 dark background로 생성되고 있다는 점**입니다. 후처리 `applyLightTheme()`가 먹히지 않았거나, 오류 때문에 중간에 중단되었거나, `uitabgroup`/tab child의 부모 배경이 dark로 유지된 것입니다.

### 402 / Video Player + AVI 제어

AVI 제어창은 상대적으로 밝아졌지만, Video Player 창은 여전히 dark bar가 강하게 남아 있습니다. 영상 자체가 검은색인 것은 허용 가능하지만, 영상 외부의 toolbar/상단 제어영역까지 검은색인 것은 샘플 기준과 맞지 않습니다.

또한 AVI 제어창의 비활성 라벨과 입력 필드의 대비가 낮습니다. `Frame:`, `Time(s):`, `Video FPS:`, `Data Hz:`, `Cache:` 등이 너무 연해서 실제 조작 가능/불가능 상태가 직관적으로 구분되지 않습니다. 버튼/입력창 높이는 이전보다 나아졌지만, 여전히 MATLAB 기본 spinner의 검은 stepper 부분이 전체 light theme와 충돌합니다.

### 403 / Map + Altitude + Video Player

메인 보드 쪽 레이아웃은 일부 개선이 보입니다. Map과 Altitude가 우측에 상하 배치되어 있고, 현재 비행 정보와 plot 데이터도 좌측/중앙에 배치되어 있습니다. 하지만 `plot 데이터`의 탭 헤더와 plot container가 여전히 어두운 색입니다. 특히 `Tab 1` 영역은 검은 배경에 어두운 글자라 시인성이 낮습니다.

Video Player가 화면 대부분을 가리므로 실제 board-off 레이아웃 검증도 어렵습니다. 이 상태에서는 레이아웃 PASS/FAIL을 판단하기보다, 비디오 창 위치/크기 기본값을 메인 검증을 방해하지 않도록 조정해야 합니다.

### 404 / Plot만 남은 화면

이 화면은 레이아웃 붕괴가 가장 잘 보입니다. 왼쪽 `현재 비행 정보`와 `plot 데이터`만 있고, 우측 대부분이 빈 흰 공간입니다. 만약 Map/Altitude/Attitude를 숨긴 상태라면 Plot 데이터가 남은 공간을 확장해 차지해야 하는데, 현재는 Plot 패널 width가 고정되어 있습니다.

이는 `applyBoardHsplit()` 또는 `reflowBoardColumns()`가 “보이는 패널 수에 따라 남은 공간을 재분배”하지 못하고 있다는 뜻입니다. 이전 [339]/[340]/[342] 문제는 일부만 처리되었고, 일반적인 panel visibility matrix 전체는 아직 미완성입니다.

### 405 / Files 탭

Files 탭도 Project 탭과 동일하게 dark content가 그대로입니다. 특히 파일 경로 표시 `(없음)`은 어두운 파란색이라 검은 배경에서 거의 보이지 않습니다. 버튼들은 우측에 몰려 있고, 본문은 대부분 빈 검은 공간입니다. 이 탭은 기능적으로도 좌우 여백 낭비가 크고, 스타일도 샘플 기준에 맞지 않습니다.

### 406 / Sync 탭

Sync 탭은 최악에 가깝습니다. 검은 배경에 회색/흰색/짙은 파란색이 섞여 있고, `Offset` 텍스트는 거의 보이지 않습니다. Numeric input의 배경도 어둡고, 버튼도 dark style입니다. 이는 `buildEditTabSync()`가 여전히 생성 시점에서 dark style 또는 parent dark background에 종속되어 있다는 뜻입니다.

또한 “Flight 1 기준 시간”, “Flight 2 기준 시간” 라벨의 잘림도 보입니다. 즉, 색상 문제뿐 아니라 grid column width도 부족합니다.

### 407 / Files 탭

405와 동일합니다. `Export everything...` 버튼은 초록색으로 튀지만, 전체 dark theme 안에서만 보이는 식이라 샘플의 light GUI와 조화되지 않습니다. 파일 선택/로드 버튼도 dark background 위에 회색 테두리라 통일성이 없습니다.

### 408 / Options 또는 Plot Manager 빈 화면

흰 빈 화면입니다. 이것은 “검은 배경 제거 성공”이 아니라 **탭 내용 생성 실패**로 보입니다. `inputBg` 오류가 발생한 시점 이후 Options 탭 build가 중단되어 내부 table/dropdown/editor가 생성되지 않았을 가능성이 큽니다. 이 상태는 스타일 개선이 아니라 기능 회귀입니다.

### 409 / Plot Manager 빈 화면

`Flight:` 라벨만 있고 나머지가 비어 있습니다. Plot Manager가 정상이라면 tree, plot 속성 패널, 버튼들이 나와야 합니다. 현재는 `buildEditTabPlotManager()` 또는 그 후속 refresh가 완료되지 않은 것으로 보입니다. 이것도 스타일 문제가 아니라 기능 회귀입니다.

## 3. 현재 패치의 핵심 문제

첫째, **theme token 관리가 깨졌습니다.** `buildEditTabOptions()`는 `inputBg`를 사용하지만 `getLightTheme()`에는 해당 필드가 없습니다. 이런 방식이면 앞으로 `tableHeaderBg`, `tabBg`, `dialogBg` 같은 필드에서도 같은 오류가 반복됩니다.

둘째, **전역 light theme 적용이 아니라 부분 후처리 방식**입니다. 검토 MD에서도 “동적으로 생성되는 UI가 많아 생성/재생성 때마다 theme 적용이 보장되어야 한다”고 지적했는데, 현재 결과는 그 문제가 그대로 나타납니다. 

셋째, **Edit Dialog는 기본 `uitabgroup` 구조를 계속 사용하고 있습니다.** 이전 검토에서도 `uitabgroup` 헤더가 계속 어둡게 나오면 custom tab bar로 교체해야 한다고 했는데, 이번 캡처에서도 Project/Files/Sync/Options/Plot Manager/Export 탭 스트립은 여전히 검은색입니다. 

넷째, **board visibility matrix가 아직 완성되지 않았습니다.** [339]/[340]/[342]만 일부 대응했을 뿐, Map/Altitude/Attitude/Plot/Info 조합별로 남는 공간을 재분배하는 일반 로직이 부족합니다. 이전 검토에서도 `applyBoardHsplit()`이 단일 구조로 blank를 하단 row에서만 해결하려고 해서 문제라고 지적했습니다. 

## 4. 우선순위별 수정 방향

**1순위는 오류 수정입니다.** `inputBg` 필드 누락을 해결하지 않으면 Options 탭과 후속 초기화가 계속 깨집니다. `getLightTheme()`에 모든 UI token을 명시하거나, `safeThemeField(t, field, fallback)` 같은 helper를 만들어 없는 필드를 fallback 처리해야 합니다.

**2순위는 Edit Dialog 재구성입니다.** Project, Files, Sync 탭이 아직 dark인 이유는 단순 theme 후처리로 해결되지 않습니다. `uitabgroup` 기본 header를 계속 쓸 것이 아니라, 상단에 `uibutton` 기반 custom tab bar를 만들고, 아래 content panel만 교체하는 구조가 필요합니다. 이 방식이면 탭 header 배경/폰트/선택상태를 완전히 제어할 수 있습니다.

**3순위는 각 buildEditTab 함수의 parent background 강제 지정입니다.** `buildEditTabProject`, `buildEditTabFiles`, `buildEditTabSync`, `buildEditTabOptions`, `buildEditTabPlotManager`, `buildEditTabExport` 시작부에서 parent, grid, panel, labels, editfields, buttons를 모두 theme token으로 통일해야 합니다. 현재처럼 일부 탭만 고치면 401/405/406처럼 탭마다 결과가 달라집니다.

**4순위는 main plot tab의 custom tab header입니다.** 메인 `plot 데이터`의 `Tab 1` 역시 검은색으로 남아 있습니다. 이것도 `uitabgroup` 기본 header 문제라면 custom tab strip으로 바꿔야 합니다.

**5순위는 layout matrix 완성입니다.** 404처럼 Map/Altitude를 끈 상태에서 Plot이 확장되지 않는 문제를 포함해, 모든 조합에서 `visible panels → ColumnWidth/RowHeight/Layout span`을 다시 계산해야 합니다.

## 판단

이 상태는 **수정 실패**로 보는 것이 맞습니다. 사용자가 요구한 것은 “blank 제거”였는데, 현재는 blank가 좌측 하단만이 아니라 **화면 우측 대부분으로 확대**되었습니다. 특히 404/401류 화면처럼 `현재 비행 정보 + plot 데이터`만 남는 조합에서는 다음처럼 되어야 합니다.

```text
현재:
[현재 비행 정보][좁은 plot 데이터][거대한 blank]

정상 기대:
[현재 비행 정보][넓은 plot 데이터              ]
```

또는 Map/Altitude가 켜진 상태라면 다음이 되어야 합니다.

```text
정상 기대:
[현재 비행 정보][넓은 plot 데이터][Map/Altitude]
```

즉, `plot 데이터`는 **항상 남는 가로 공간을 흡수하는 primary flexible panel**이어야 합니다. 지금처럼 작은 width로 고정되면 안 됩니다.

## 수정 방향

`reflowBoardColumns()`와 `applyBoardHsplit()`에서 다음 원칙을 강제해야 합니다.

### 1. Plot 데이터는 기본 `'1x'` flexible column으로 둔다

`panelDataView`가 visible이면 해당 column은 반드시 `'1x'` 또는 그 이상 flex를 가져야 합니다. 다른 패널이 꺼질수록 plot column이 확장되어야 합니다.

예상 column 정책은 다음과 같습니다.

```text
Info + Plot만 visible:
ColumnWidth = {infoFixed, splitter, '1x'}

Info + Plot + MapAlt visible:
ColumnWidth = {infoFixed, splitter, '1x', splitter, mapAltFixed}

Plot만 visible:
ColumnWidth = {'1x'}

Info + Plot + Attitude visible:
ColumnWidth = {infoFixed, splitter, '1x', splitter, attitudeFixed}
```

현재처럼 plot을 고정 column에 두고 나머지 column을 0 또는 blank로 남기면 안 됩니다.

### 2. 숨겨진 panel의 column은 0으로 만들고, 남는 공간은 plot으로 넘긴다

Map/Altitude가 off이면 그 column width는 0이어야 하고, 그만큼 plot이 넓어져야 합니다. Attitude가 off여도 동일합니다. “숨김”은 단순 `Visible='off'`가 아니라 **column width 재분배**까지 포함해야 합니다.

### 3. board-off 전용 hsplit에서도 plot은 row span + width 확장을 가져야 한다

이전 [339]/[340]/[342] 요구와 연결됩니다. `panelDataView`가 `Row=1, Column=3`에만 머물면 height와 width 둘 다 부족합니다. 최소한 다음처럼 가야 합니다.

```text
Case: info + plot만 실질 visible
info: Row [1 3], Column 1
plot: Row [1 3], Column 3, Width '1x'

Case: info + plot + mapAlt
info: Row [1 3], Column 1
plot: Row [1 3], Column 3, Width '1x'
mapAlt: Row [1 3], Column 5, fixed width, vertical
```

이전 검토 문서도 `panelDataView`가 상단 row에만 있어 세로 확장이 안 된다고 지적했고, 필요한 것은 조건별 layout matrix라고 정리했습니다.

## 다음 패치에서 반드시 들어가야 할 구체 요구

다음 수정에는 `plot 데이터 width` 문제를 별도 최우선 항목으로 추가해야 합니다.

```text
Fix plot data panel width allocation.

In all main dashboard layouts, panelDataView must be the primary flexible panel.
It must absorb all remaining horizontal space after visible fixed-width panels
(info, map/altitude, attitude) are allocated.

Do not leave large unused blank regions to the right of panelDataView.

Required behavior:

4. If attitude is hidden, its previous width must be transferred to plot or to the intended visible panel according to the layout matrix.
5. In board-off hsplit mode, panelDataView must use Row [1 3] and a flexible '1x' column whenever it is visible.
6. No visible layout state should show a large blank region to the right of plot data.
```
