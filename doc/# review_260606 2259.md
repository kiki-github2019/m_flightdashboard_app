# review 1
검토 결과, 이번 수정은 **일부 개선은 있었지만 핵심 요구사항에는 아직 미달**입니다. 특히 “샘플.png처럼 검은색 배경이 없는 light GUI”라는 최상위 요구가 아직 코드와 실행 결과에 모두 관철되지 않았습니다. 제 판단으로는 **적합도 45~50/100 수준**입니다.

## 1. 전체 판단

수정 전보다 좋아진 부분은 있습니다. 상단 헤더와 메인 배경은 이전보다 밝아졌고, 비행 자세 패널의 외부 숫자 표시도 이전보다 읽기 쉬워졌습니다. `setMapAltArrangement`, `applyBoardHsplit`, `applyLightTheme`, `applyThemeToPanels`, `applyThemeToAxes` 같은 함수가 추가·보강되어 방향 자체는 맞습니다. 다만 현재 결과는 **“기능별 부분 패치”에 가깝고, GUI 전체를 샘플 기준으로 재정의한 수준은 아닙니다.** 코드에서도 theme dispatcher는 존재하지만, dark/blue 계열을 조건부로만 보정하고 동적 생성 UI까지 일괄 보장하지 못하는 구조입니다. 

가장 큰 문제는 **검은색/진한 회색이 아직 핵심 영역에 남아 있다는 점**입니다. 실행 캡처에서 `plot 데이터` 탭 헤더, Plot Manager/Options 내부 하단 영역, Edit Dialog 탭 스트립, Video Player 상단/외곽 영역이 여전히 어둡습니다. 이는 이전 요청의 핵심인 “샘플에는 어디도 검은색 배경이 없음”과 정면으로 충돌합니다.

## 2. 실행 화면 기준 문제점

첫 번째 캡처에서 가장 심각한 부분은 **설정/프로젝트 편집기**입니다. 창 외곽과 일부 본문은 밝아졌지만, `Project / Files / Sync / Options / Plot Manager / Export` 탭 헤더가 여전히 검은색 계열입니다. 또한 `RequiredColumns` 탭 아래쪽에는 거대한 검은색 빈 영역이 남아 있습니다. 이 영역은 기능적으로도 낭비이고, 시각적으로도 샘플 GUI와 전혀 맞지 않습니다. “검은 배경 제거” 요청이 핵심이었는데, 가장 눈에 띄는 dialog 중앙부에 그대로 남아 있어 결과가 실망스러울 수밖에 없습니다.

두 번째 캡처의 **Video Player / AVI 제어**도 부분 개선에 그쳤습니다. AVI 제어창 본문은 밝아졌지만, 비활성 라벨이 지나치게 연하고 버튼 일부는 여전히 세로 여유가 부족해 보입니다. Video Player 쪽은 영상 자체가 검은 배경인 것은 허용될 수 있지만, 영상 외부의 상단 제어 바와 창 헤더가 어두운 톤이라 “샘플 기반 light theme” 원칙과 충돌합니다. 영상 컨텐츠가 검은색인 것과, GUI 컨테이너가 검은색인 것은 구분해야 하는데 현재는 이 경계가 명확하지 않습니다.

메인 화면에서는 `plot 데이터` 영역의 탭 헤더와 plot 주변 배경이 여전히 어둡습니다. plot 자체의 축 배경은 밝아진 부분이 있지만, 탭 라벨 주변이 검은색이라 사용자가 보는 첫인상은 여전히 “검은 테마가 남아 있다”입니다. 이 문제는 `applyThemeToAxes`만으로는 해결되지 않습니다. 축 내부가 아니라 **tab group / tab header / plot container** 문제이기 때문입니다.

## 3. 코드 구조상 근본 원인

현재 코드는 `getLightTheme()`와 `applyLightTheme()`를 추가해 전역 테마처럼 보이게 만들었지만, 실제 적용 방식은 “이미 만들어진 컴포넌트를 나중에 찾아서 보정”하는 후처리 방식입니다. `applyLightTheme()`는 panels, buttons, labels, tables, axes, inputs, tabs, trees를 순서대로 보정하지만, MATLAB UI 컴포넌트 중 일부는 이 방식으로 완전히 제어되지 않습니다. 특히 `uitabgroup`의 탭 헤더, 일부 `uitable` 내부 영역, `uitree`, OS/Java 기반 창 title bar는 단순 `BackgroundColor` 지정만으로 원하는 색이 나오지 않을 수 있습니다. 

또한 `applyThemeToPanels()`의 주석과 로직을 보면 “blue 의도는 보존”하고 “near-black non-video만 light normalize”하는 방식입니다. 즉, 검은색과 어두운 파란색을 **무조건 금지**한 것이 아니라, 일부 색은 의도된 색으로 간주하고 남겨둘 여지가 있습니다. 사용자 요청은 “샘플처럼 검은색 배경 제거”였으므로, 현재 로직은 요구사항보다 약합니다. 

`applyThemeToTabs()`도 한계가 큽니다. 코드는 `uitab`의 `BackgroundColor`, `ForegroundColor`를 바꾸려 하지만, R2023b의 실제 렌더링에서 탭 헤더가 원하는 대로 바뀌지 않는 경우가 있습니다. 캡처에서 검은 탭 헤더가 남아 있는 것으로 보아, 이 부분은 단순 theme 함수가 아니라 **custom tab header를 직접 만드는 방식**으로 우회해야 합니다. 예를 들어 `uitabgroup` 기본 탭 헤더를 계속 쓰지 말고, 상단에 `uibutton` 기반 탭 바를 만들고 아래 콘텐츠 패널만 교체하는 방식이 필요합니다.

## 4. Board-off / Map / Altitude 관련 평가

코드에는 `setMapAltArrangement()`가 추가되어 horizontal/vertical 전환을 시도합니다. `horizontal`일 때 map과 altitude를 1행 2열로 배치하는 로직은 들어갔고, `applyMapAltVisibility()`에서 board-off source일 때 horizontal을 선택하도록 되어 있습니다. 방향은 맞습니다. 

그러나 `applyBoardHsplit()`은 여전히 3-row 구조, 즉 upper/info+plot, splitter, lower/attitude+map-alt 구조를 기본으로 사용합니다. 코드상으로는 lower-left blank 제거를 의도한 분기들이 보이지만, 실제 캡처에서는 board-off 시나리오 전체가 충분히 검증된 것으로 보기 어렵습니다. 특히 사용자가 요구한 “하단 보드 off일 때 Map/Altitude는 좌우 배치”, “현재 비행 정보가 blank 영역까지 확장”, “Plot 데이터는 height를 길게 확보” 같은 조건별 matrix가 명확히 검증되지 않았습니다. 

즉, board-off는 **구현 시도는 있었지만, 검증 완료 상태는 아닙니다.** Claude Code에 다시 지시할 때는 “함수 추가 여부”가 아니라 “각 캡처 조건별 실제 화면 PASS/FAIL”을 요구해야 합니다.

## 5. Gauge 개선 평가

Gauge는 이전보다 개선되었습니다. Pitch/Roll/Heading 외부 숫자 폰트가 커졌고, 중복된 내부 흰색 숫자 표시는 많이 정리된 것으로 보입니다. `createGaugePanel()`에서도 label font size를 15로 키우고, gauge panel 배경을 theme 기반으로 두려는 시도가 있습니다. 

하지만 아직 완성은 아닙니다. 원형 게이지 자체의 tick label은 여전히 작고, 바늘/눈금/외부 값의 시각 계층이 명확하지 않습니다. 현재는 “숫자는 읽히지만, 고급 GUI처럼 정돈된 상태”는 아닙니다. 샘플 스타일을 따르려면 gauge 내부는 더 단순화하고, 외부 값 label을 카드형 정보로 명확히 분리하는 것이 좋습니다.

## 6. RequiredColumns 문제

`RequiredColumns`는 여전히 요구사항을 충분히 만족하지 못합니다. 현재 구현은 `EDOptReqTable`의 table cell edit을 받아 `draft.mappedCols.(key) = char(evt.NewData)`로 저장하는 방식입니다. 즉, 사용자가 직접 문자열을 입력하거나 table 값을 바꾸는 구조입니다. 

사용자 요청은 “Column 항목을 선택하면 로드된 비행데이터 항목들이 표시되고 다른 항목으로 변경 가능”입니다. 이 요구는 단순 editable table이 아니라, 선택된 row에 대해 **로드된 데이터 컬럼 목록을 dropdown/listbox로 보여주고 선택하도록 하는 UX**입니다. 현재 방식은 오타, 존재하지 않는 column명, 대소문자 불일치, 잘못된 매핑을 유발할 수 있습니다. 검증 버튼이 있더라도 UX 자체가 요구사항과 다릅니다.

## 7. 왜 계속 실패하는가

핵심 원인은 Claude Code가 “색 몇 개 수정 + 보정 함수 추가” 방식으로 접근했기 때문입니다. 이번 요구는 그런 수준이 아니라 **GUI styling policy를 전역 설계로 바꾸는 작업**입니다.

현재 방식의 문제는 다음과 같습니다.

첫째, `getLightTheme()` 값은 있지만 모든 생성 코드가 그 theme token만 쓰지는 않습니다. 여전히 곳곳에 직접 색상값이 있고, 일부는 나중에 후처리로 보정하려고 합니다.

둘째, 동적으로 생성되는 UI가 많습니다. plot tab, edit dialog refresh, video dialog, table, tree, axes 등이 생성·재생성될 때마다 같은 theme가 재적용되어야 하는데, 현재는 일괄 보장이 약합니다.

셋째, MATLAB R2023b 기본 UI 컴포넌트의 스타일 한계를 고려하지 않았습니다. `uitabgroup` 헤더가 계속 어둡게 나오면, `BackgroundColor`를 더 바꾸는 것이 아니라 기본 탭 헤더 사용을 포기해야 합니다.

넷째, “샘플 기반”이라는 정성적 요구가 코드상 검증 조건으로 바뀌지 않았습니다. Claude Code가 완료 보고를 할 때 “dark background zero check”, “screenshot visual checklist”, “findall 기반 dark component dump” 같은 검증을 하지 않으면 같은 문제가 반복됩니다.

## 8. 다음 수정 방향

다음 단계는 부분 패치가 아니라 **강제 light-theme audit**로 가야 합니다.

가장 먼저, 앱 실행 후 `findall(app.UIFigure)`와 `findall(app.EditDialog)`를 대상으로 `BackgroundColor`, `Color`, `ForegroundColor`, `FontColor`를 모두 스캔해서 평균 밝기 0.45 미만의 UI 요소를 리포트하도록 해야 합니다. Video frame axes처럼 예외가 필요한 요소만 whitelist에 넣고, 나머지는 전부 수정 대상으로 삼아야 합니다.

두 번째로, `uitabgroup` 기본 탭 헤더가 계속 검게 나오면 custom tab bar로 교체해야 합니다. 특히 메인 `plot 데이터` 탭과 설정/프로젝트 편집기의 Project/Files/Sync/Options/Plot Manager/Export 탭은 현재 문제의 핵심입니다.

세 번째로, Edit Dialog의 Options/Plot Manager 내부는 기본 컴포넌트에 맡기지 말고 `uigridlayout + uipanel + uibutton tab bar` 구조로 다시 구성해야 합니다. 지금처럼 `uitabgroup` 안에 또 `uitabgroup` 또는 `uitable/tree`를 넣으면 R2023b 렌더링에서 어두운 영역이 계속 남을 가능성이 큽니다.

네 번째로, RequiredColumns는 table cell edit 방식이 아니라 “선택 row + 오른쪽 dropdown editor”로 바꿔야 합니다. dropdown의 Items는 `rawDataUnscaled.Properties.VariableNames`에서 가져와야 하고, 선택 즉시 draft에 반영되어야 합니다.

마지막으로, 수정 완료 기준을 “코드상 적용”이 아니라 “실행 캡처 기준”으로 둬야 합니다. 현재처럼 코드에 `v3-sample: black 제거` 같은 주석이 있어도 실제 화면에 검은 영역이 남으면 FAIL입니다.

## 9. 결론

이번 수정은 **방향은 맞지만 결과는 아직 실패에 가깝습니다.** 특히 사용자가 가장 강하게 요청한 “샘플 기반 light style, 검은색 배경 제거, 배경색과 폰트색 조화”가 실행 화면에서 명확히 깨졌습니다. 지금 필요한 것은 추가 색상 패치가 아니라, **dark UI 요소를 자동 검출하고, 기본 탭/트리/테이블 렌더링 한계를 우회하는 구조적 수정**입니다.

## 결론

최신 코드의 `applyBoardHsplit()`은 기본적으로 다음 구조입니다.

```text
Row 1: 현재 비행 정보       | Plot 데이터
Row 2: splitter
Row 3: 비행 자세 또는 Map/Alt | Map/Alt 또는 빈 영역
```

즉, 현재 구현은 **상단 row와 하단 row를 분리한 H-split 구조**입니다. 하지만 이전 요청의 핵심은 “blank 영역을 정보/지도/고도/plot이 상황별로 다시 차지하도록 배치”하는 것이었습니다. 현재 코드는 그 요구를 조건별로 정확히 구현하지 못했습니다.

## 코드상 문제점

`reflowBoardColumns()`는 board-off source 보드일 때 무조건 `applyBoardHsplit(fIdx)`로 진입합니다. 이 자체는 board-off 전용 재배치를 하겠다는 방향은 맞습니다. 

하지만 `applyMapAltVisibility()`에서는 board-off source일 때 `setMapAltArrangement(fIdx, 'horizontal')`을 호출합니다. 이는 [339]의 “Map, Altitude는 우측에 상하로 배치” 요구와 충돌합니다. `setMapAltArrangement()`도 horizontal이면 Map/Altitude를 1행 2열로 배치하고, vertical일 때만 상하 배치를 사용합니다. 

더 큰 문제는 `applyBoardHsplit()`의 패널 배치입니다. 최신 코드에서는 `panelInfo`를 항상 `Row=1, Column=1`, `panelDataView`를 항상 `Row=1, Column=3`에 둡니다. 즉, “현재 비행 정보가 좌측 하단 blank까지 차지”하거나 “Plot 데이터가 상하로 길게 배치”되는 구조가 아닙니다. 

또한 attitude가 꺼지고 Map/Altitude가 켜진 경우에는 `panelMapAlt`를 `Row=3, Column=[1 3]`로 하단 전체에 배치합니다. 이는 “Map만 blank 영역에 표시” 또는 “Altitude만 blank 영역에 표시”라는 [340], [342] 요구와 다릅니다. blank 영역만 채우는 것이 아니라 하단 전체를 차지할 수 있고, 그 결과 Plot 데이터가 세로로 확장되지 않습니다.

```text
현재 비행 정보: 좌측 전체 높이 또는 blank 포함 확장
Plot 데이터: 중앙/우측에서 세로로 길게
Map + Altitude: 우측 고정 폭, 상하 배치
```

현재 코드는 다음에 가깝습니다.

```text
상단: 현재 비행 정보 | Plot 데이터
하단: 비행 자세     | Map/Altitude
```

따라서 [339]는 **거의 미반영**입니다. 특히 `panelInfo`와 `panelDataView`에 row-span이 없습니다.

### [340] 수정12

요구는 “Map을 좌측 하단 blank 영역에 표시하고, Plot 데이터는 상하로 길게”입니다.

현재 코드는 attitude가 꺼지고 map이 켜졌을 때 `panelMapAlt`를 하단 `Column=[1 3]`로 확장합니다. 이는 blank 영역 활용 의도는 있으나, “Map만 좌측 하단 blank에 넣고 Plot을 세로 확장”하는 요구와 다릅니다. **부분 반영이지만 레이아웃 목표는 불일치**입니다.

### [342] 수정13

요구는 “Altitude를 좌측 하단 blank 영역에 표시하고, Plot 데이터는 상하로 길게”입니다.

현재 코드는 Altitude 단독도 `panelMapAlt` 컨테이너 안에서 처리합니다. 단독 가시일 때 `setMapAltArrangement()`가 fill 처리하긴 하지만, board-off H-split 구조에서는 여전히 `panelDataView`가 상단 row에만 있으므로 Plot 데이터의 세로 확장이 없습니다. **부분 반영이나 핵심 미반영**입니다.

## 핵심 원인

문제는 `applyBoardHsplit()`이 “blank 제거”를 **하단 row 안에서만 해결**하려고 했기 때문입니다. 하지만 이전 요청은 단순히 하단 blank를 없애는 것이 아니라, 상황별로 다음을 요구했습니다.

1. 정보 패널이 아래쪽 blank까지 확장한다.
2. Plot 패널은 세로로 길게 확장한다.
3. Map/Altitude는 상황에 따라 우측 상하 또는 좌측 하단 blank에 배치한다.

즉, 필요한 것은 단일 H-split 함수가 아니라 **조건별 layout matrix**입니다.

## 수정 방향

`applyBoardHsplit()`을 다음처럼 조건 분기해야 합니다.

```text
Case 339: info + plot + map + altitude visible, attitude hidden or board-off blank 발생
- info: Row [1 3], Column 1
- plot: Row [1 3], Column 3
- mapAlt: Row [1 3], Column 5 또는 별도 right column
- mapAlt 내부: vertical, Row 1 = Map, Row 2 = Altitude

Case 340: map only visible, attitude hidden
- map: blank area 또는 left-lower target cell
- plot: Row [1 3]로 세로 확장
- info: 필요 시 left/full 또는 기존 위치 유지

Case 342: altitude only visible, attitude hidden
- altitude: blank area 또는 left-lower target cell
- plot: Row [1 3]로 세로 확장
```

현재처럼 `panelInfo = row1`, `panelDataView = row1`, `panelMapAlt = row3`로 고정하면 이전 요청은 계속 실패합니다.

최종 판단은 **“코드에 blank 제거를 시도한 흔적은 있으나, [339], [340], [342]의 구체 레이아웃 요구는 반영되지 않았다”**입니다.
